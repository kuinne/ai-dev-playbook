# YApi API 代码生成

## 使用场景

当出现以下类似请求时，应用此 skill：

- “根据这个地址给我接口定义”
- “根据这个 YApi 文档生成前端 API 和类型定义”
- “帮我把 YApi 文档里的接口转成项目里的 API 调用代码”
- “YApi 文档有更新，帮我同步到现有的接口和类型里”
- 任何提到 **YApi / 接口定义 / API 类型 / yapi-auto-mcp** 的需求

目标是：在保持项目既有代码和类型风格的前提下，从 YApi 文档生成或**增量更新**可直接放入前端项目的 **API 函数**、**类型定义** 代码，并在用户需要时生成可切换的 **mock 分支**。

---

## 总体工作流概览

1. 识别当前项目中已有的 API 定义、HTTP 封装和类型定义的风格与位置（如果存在）
2. 使用 `yapi-auto-mcp` MCP 服务解析用户提供的 YApi 地址，获取结构化接口信息
3. 基于接口描述和字段说明设计业务化的函数名、类型名和枚举
4. 生成带有 JSDoc 注释的 API 函数和类型定义代码，尽量贴合项目现有风格
5. 询问用户是否需要 mock，如需要则为接口生成基于类型推导的 mock 数据与 mock 分支
6. 按模块/标签组织输出结构，并与用户确认细节（命名、文件路径、mock 方式等）

---

## 步骤 1：识别项目 API 与类型风格

1. **定位候选文件/目录**：使用 `Glob` / `Grep` 在项目中查找常见 API 与类型封装位置，例如：
   - `src/api/**`
   - `src/services/**`
   - `src/apis/**`
   - `src/request/**`
   - `src/types/**`
   - `src/**/types.ts`
   - `http-client.ts`、`request.ts`、`client.ts` 等

2. **从已有代码中归纳 API 与类型风格**（如果找到）：
   - 确认使用 **TypeScript 还是 JavaScript**
   - 确认 HTTP 调用方式：
     - 原生 `fetch`
     - `axios`
     - 自封装的 `http-client` 工具（如 `httpClient.get<T>(...)`）
   - 观察 API 函数命名约定：
     - 例如 `getUserDetail` / `createOrder` / `updateUserProfile`
   - 观察类型命名与组织方式：
     - 是否按接口划分：`GetUserDetailParams` / `GetUserDetailResult`
     - 是否抽出业务实体：`UserInfo` / `OrderItem`
     - 类型文件的存放位置：例如：
       - 与 API 同文件（如 `user-api.ts` 内同时定义 API 和类型）
       - 独立 `types` 目录（如 `src/types/user.ts`）
       - 每个模块一个 `types.ts`
   - 观察类型导出风格：
     - `type` vs `interface`
     - 默认导出还是具名导出
   - 观察 JSDoc 风格：
     - 是否为每个函数和类型写 `/** ... */` 注释
     - 是否使用 `@param` / `@returns` 标签

3. **如果没有明显的 API/类型风格可参考**，必须主动向用户询问至少以下信息：
   - 是否优先使用 **TypeScript** 生成（可通过检测 `tsconfig.json` 或 `.ts/.tsx` 占比来辅助判断，默认优先 TS）
   - 期望的 HTTP 封装模式（例如“按当前项目的 `http-client.ts` 来”、“用 axios 封装成 `request({ url, method, data })` 风格”等）
   - **类型定义放置策略**：
     - 与 API 同文件（例如每个 `xxx-api.ts` 内包含 API 与类型）
     - 独立类型目录，如 `src/types/` 或 `src/domain/`
     - 其它自定义路径（由用户说明）
   - 希望将生成的 API 和类型放在哪个目录（例如 `src/api/yapi/`、`src/services/`）

---

## 步骤 2：通过 yapi-auto-mcp 解析 YApi 文档

1. 首先 **阅读 MCP 工具描述**：
   - 使用 `CallMcpTool` 前，先读取 `yapi-auto-mcp` 的工具 schema（位于对应 MCP 目录下的 `tools/*.json`），了解：
     - 可用的工具名称
     - 每个工具的参数（如 YApi 地址、项目 id、token 等）
     - 返回数据结构

2. 根据用户提供的 YApi 地址，选择合适的工具调用，例如：
   - 输入：接口分组地址 / 项目地址 / 单个接口地址
   - 输出：包含以下信息的结构化数据（按工具实际返回为准）：
     - 接口路径（`/api/user/detail`）
     - HTTP 方法（GET/POST/PUT/DELETE 等）
     - 接口名称 / 标题 / 简要说明
     - 请求参数（query/body/path/header）字段及其描述
     - 响应结构和字段说明
     - 分组/标签信息（用于模块划分）

3. 将 MCP 返回的数据整理为内部统一模型，方便后续生成代码时使用。

## 步骤 3：类型建模规则

对每个 YApi 接口，按以下规则生成类型定义：

1. **请求参数类型**：
   - 将 query/path/body 等参数建模为类型：
     - 若项目已有统一模式（例如所有参数合并为一个 `Params` 类型或按模块集中管理），优先遵循项目约定（包括类型放在哪个文件）
     - 否则：
       - 为接口生成一个 `XXXParams` 类型，字段来源于 YApi 参数定义
   - 字段命名与 YApi 保持一致，如有不合法标识符（带 `-` 等），转换为合法 camelCase，并在 JSDoc 中保留原始名称说明。

2. **响应类型**：
   - 如果项目已有统一响应包装（例如 `ApiResponse<T>`），则将业务数据部分建模为 `XXXResult`，并在函数返回类型中包一层：
     - `Promise<ApiResponse<GetUserDetailResult>>`
   - 否则，将整个响应结构建模为 `XXXResponse` 或 `XXXResult` 类型。

3. **业务实体类型**：
   - 当多个接口复用同一结构时（例如用户、订单），抽取为独立业务类型：
     - `UserInfo`、`OrderItem` 等
   - 优先从接口名称/字段描述中提取业务含义来命名，而不是 `Data`、`Result` 这类泛型名。
   - 若项目已有对应实体类型，优先尝试复用并在必要时扩展，而不是重复定义。

4. **枚举与字面量类型**：
   - 当字段描述中出现类似「0：禁用，1：启用」「1-待支付，2-已支付，3-已取消」的文字时：
     - 优先生成业务含义清晰的 **枚举类型** 或字面量联合类型：
       - `enum UserStatus { Disabled = 0, Enabled = 1 }`
       - 或 `type OrderStatus = 'PENDING' | 'PAID' | 'CANCELLED'`
     - 在 JSDoc 中注明各枚举值的含义
   - 枚举或联合类型命名规则：
     - 基于字段名称和业务含义，例如 `UserStatus`、`OrderStatus`，避免 `StatusEnum` 这类缺乏上下文的名字。

5. **JSDoc 注释**：
   - 对每个导出的类型添加 JSDoc：
     - 简要说明来自 YApi 的描述
     - 对于复杂类型，可在注释中补充字段含义摘要

## 步骤 4：API 函数生成与 mock 规则

### 4.1 函数命名

1. 基于接口名称和业务含义生成函数名，例如：
   - 「获取用户详情」 → `getUserDetail`
   - 「创建订单」 → `createOrder`
   - 「更新用户头像」 → `updateUserAvatar`
2. 避免无业务含义的通用命名，如 `requestApi`、`doFetch`、`postData` 等。

### 4.2 函数签名

1. 若项目已有统一封装（如 `httpClient`），则沿用其调用方式，例如：
   - `export function getUserDetail(params: GetUserDetailParams): Promise<ApiResponse<GetUserDetailResult>>`
2. 若无明确封装，使用通用但清晰的结构，例如（以 TS 为例，可按项目实际风格调整）：
   - `export async function getUserDetail(params: GetUserDetailParams): Promise<GetUserDetailResult>`

### 4.3 JSDoc 注释

1. 从 YApi 接口的标题和描述中提取信息，生成函数注释：
   - 概要说明接口作用
   - 使用 `@param` 和 `@returns` 描述主要参数和返回结果
2. 标注：
   - 重要字段的业务含义
   - 特殊约束（例如“只对管理员可用”、“分页参数从 1 开始”）

### 4.4 mock 支持

在为接口生成 API 函数时，必须按照以下流程处理 mock：

1. 在开始生成具体代码前，**询问用户是否需要为本次生成的接口提供 mock 能力**，例如：
   - “是否需要为这些接口生成基于类型的 mock 数据并在函数中通过 `mock` 标志控制？”
2. 若用户回答 **不需要 mock**：
   - 只生成真实接口调用代码，不包含 mock 分支。
3. 若用户回答 **需要 mock**：
   - 为每个接口基于其请求/响应类型推导一个合理的 **mock 数据结构**，例如：
     - 字符串字段使用示例值（可来自字段描述）
     - 数值字段使用示例数值（如 `1`、`100`）
     - 布尔值字段使用 `true` 或依据字段名称推断
     - 数组字段生成包含 1–2 个元素的示例数组
   - 在 API 函数中增加一个可控制 mock 行为的标志（根据项目风格选择其形式）：
     - 例如在参数中新增可选配置：`options?: { mock?: boolean }`
     - 或在函数内部读取一个约定的配置常量（如 `const useMock = false`），但应向用户说明并允许其调整。
   - 函数逻辑中必须包含 **显式的 mock 分支**：
     - 当 `mock` 标志为 `true` 时：
       - 直接返回事先构造好的 mock 数据（必要时使用 `Promise.resolve(mockData)` 保持返回类型一致）
     - 当 `mock` 标志为 `false`（或未设置）时：
       - 发起真实 HTTP 请求，按项目既有封装方式返回真实数据。
   - 在 JSDoc 注释中说明 mock 行为：
     - 指明 `mock` 标志的含义与默认值
     - 简要描述 mock 数据的用途（如“用于本地开发/联调时无需真实后端”）

## 步骤 5：输出结构与与用户交互

1. **文件与模块划分**：
   - 优先按 YApi 的分组/标签进行拆分：
     - 例如 `user-api.ts`、`order-api.ts`、`auth-api.ts`
   - 如项目已有更明确的目录结构（例如 `src/services/user.ts`），优先遵循项目现有方案。
   - 对于类型定义的位置：
     - 若项目中已有统一类型组织方案（例如集中在 `src/types/`），优先放入对应位置
     - 若没有明显方案，先询问用户偏好（与 API 同文件 / 单独 `types` 目录 / 其它路径），再按用户选择输出。

2. **输出给用户时**：
   - 先简要说明：
     - 本次生成覆盖了哪些模块（如“用户相关接口”、“订单相关接口”）
     - 每个模块文件里包含哪些函数和类型
     - 是否为这些接口启用了 mock 分支，以及如何切换 mock/真实调用
   - 给出一到两个代表性接口的代码片段，方便用户快速确认风格是否符合预期。

3. **与用户确认可调整的点**：
   - 函数命名是否符合业务习惯
   - 类型名是否需要更贴近领域术语
   - 文件拆分和输出路径是否需要调整
   - mock 方案（标志名、默认值、mock 数据形式）是否需要调整

4. **必要时的迭代**：
   - 若用户觉得命名过于通用或不够贴近业务，优先调整命名规则并重新生成，而不是手动零散改动。
   - 若用户对 mock 结构或控制方式不满意，优先统一调整生成策略，再整体重新生成。

## 步骤 6：已有代码的增量更新（YApi 变更场景）

本步骤用于处理“接口/类型已经在项目中存在，但 YApi 文档有更新，需要同步到现有代码”的场景。

1. **识别用户意图为“更新/同步”而不是“全量新生成”**：
   - 如果用户表述中包含“更新”、“同步”、“对齐 YApi”、“按最新 YApi 调整”等关键词，应进入 **增量更新模式**。
   - 在增量更新模式下，优先在现有文件中 **就地更新类型与 JSDoc**，避免无谓的重命名和大范围重写。

2. **建立 YApi 接口与现有代码的映射关系**：
   - 优先通过 **请求路径 + HTTP 方法** 来匹配：
     - 例如：`GET /api/user/detail` → 现有的 `getUserDetail` 函数及其相关类型。
   - 如果项目中已经有显式标记（例如在 JSDoc 中保留了 YApi 链接或接口 id），应优先使用这些标记来建立更加可靠的映射。
   - 映射成功后，对应关系包括：
     - API 函数名
     - 请求参数类型（如 `GetUserDetailParams`）
     - 响应/结果类型（如 `GetUserDetailResult`）
     - 相关业务实体和枚举类型（如 `UserInfo`、`UserStatus`）。

3. **类型的增量更新策略**：
   - **新增字段**：
     - 当 YApi 增加了请求参数或响应字段时：
       - 在对应类型中新增字段，默认可以设置为可选（或按项目风格决定），并附上来自 YApi 的描述 JSDoc。
   - **字段删除或不再使用**：
     - 默认不直接物理删除已有字段，避免破坏现有业务代码。
     - 可以在 JSDoc 中标记为“YApi 已移除/不再返回”，或提示用户是否需要一并清理。
   - **字段类型变更**：
     - 不应静默替换为不兼容的新类型。
     - 优先策略：
       - 向用户简要说明差异（例如“YApi 将某字段从 number 改为 string”），
       - 视情况选择：
         - 使用联合类型（如 `string | number`）做兼容，
         - 或在用户确认后再替换为新类型。
   - **枚举/字面量更新**：
     - 当 YApi 为某字段增加/减少了枚举值：
       - 在对应的枚举或字面量联合类型中增减成员，
       - 同步更新 JSDoc，保持含义与 YApi 描述一致。

4. **API 函数的增量更新策略**：
   - **尽量保持函数名和导出接口不变**：
     - 若 YApi 的语义变化不大，只是字段增减或描述更新，不要随意重命名现有函数和类型。
   - **参数与返回值类型的更新**：
     - 在保持函数签名形式不变的前提下，更新使用到的类型定义（例如 `GetUserDetailParams` 中新增字段）。
     - 如需在参数中新增配置（例如新增的 `mock` 相关选项），要保证兼容性（通过可选参数或 options 对象）。
   - **函数实现逻辑**：
     - 不主动重写用户已有的实现逻辑（包括错误处理、日志、埋点等）。
     - 仅在调用层面（如 URL、查询参数构造）确实与 YApi 新规范不符时，提出修改建议或给出更新后的代码片段，并清楚标明哪些是“需要替换的部分”。

5. **与用户的交互说明（更新模式）**：
   - 在输出结果时，明确标注：
     - 哪些接口/类型是**新增**的；
     - 哪些是**基于现有代码做了增量更新**（并简要说明更新点，如“新增字段 xxx”、“补充枚举值 yyy”）；
     - 哪些地方存在潜在的不兼容变更，需要用户特别留意。
   - 对于可能引入破坏性变更的修改（例如字段类型发生根本性变化），要：
     - 先文字说明变更原因，
     - 再给出推荐的更新代码，
     - 由用户决定是否应用或进一步调整。

6. **避免全量覆盖的注意事项**：
   - 除非用户明确要求“重新生成一份全新的文件”，在更新场景中不要：
     - 直接覆盖整个已有文件，
     - 主动删除已经存在但 YApi 中暂时缺失的业务字段，
     - 随意更改导出名称（函数名、类型名）。
   - 对于不确定如何处理的差异，要优先选择：
     - 通过注释或说明向用户暴露差异，
     - 再征求用户决策，而不是单方面做破坏性修改。
